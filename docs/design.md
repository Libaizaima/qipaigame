# 棋牌游戏网站设计文档

## 1. 项目概述

本项目计划开发一个前后端分离的网页游戏平台，第一阶段聚焦于两个游戏：

- 百家乐
- 骰子游戏

平台支持账号注册登录、网页进入游戏、虚拟筹码下注、开奖记录查询、用户下注历史查询等基础能力。

为降低第一阶段的实现复杂度与上线风险，当前版本以“娱乐版 / 虚拟筹码版”作为设计前提，不包含充值、提现、法币结算等能力。

## 2. 目标与范围

### 2.1 第一阶段目标

第一阶段要完成以下核心链路：

1. 用户注册、登录、退出登录
2. 用户进入游戏大厅
3. 用户查看百家乐和骰子两个游戏入口
4. 用户进入房间并参与下注
5. 系统完成发牌或开奖
6. 系统完成结算并更新用户余额
7. 用户查看自己的下注记录和开奖历史

### 2.2 暂不纳入第一阶段的功能

以下功能不进入首版 MVP：

- 充值 / 提现
- 真人荷官或视频直播
- 多语言国际化
- 推广代理系统
- 复杂会员等级体系
- 风控黑名单自动化策略
- 多币种钱包
- 多桌并发赛事玩法

## 3. 产品定位

本平台定位为网页端休闲棋牌游戏平台，面向通过浏览器访问的用户。产品体验重点如下：

- 登录流程简单
- 游戏大厅直达
- 游戏结果实时同步
- 下注反馈及时
- 账变记录清晰

## 4. 用户角色

### 4.1 普通玩家

可使用功能：

- 注册与登录
- 查看余额
- 进入游戏房间
- 下注
- 查看开奖结果
- 查看历史记录

### 4.2 管理员

可使用功能：

- 管理游戏房间
- 查看用户信息
- 查看下注记录与账变记录
- 配置部分游戏参数
- 临时关闭房间或维护游戏

## 5. 技术架构

## 5.1 推荐技术栈

为了兼顾开发效率、后期扩展性与前后端统一语言，推荐如下技术方案：

- 前端：Next.js + React + TypeScript
- 后端：NestJS + TypeScript
- 数据库：PostgreSQL
- 缓存与实时状态：Redis
- 实时通信：WebSocket
- 部署：Docker + Nginx
- ORM：Prisma

### 5.2 架构原则

- 前后端分离
- 游戏状态与业务状态服务端权威
- 所有余额变化必须可审计
- 游戏局数据、下注数据、账变数据分离存储
- 关键流程支持事务与幂等

### 5.3 系统模块图

```text
Web Frontend
  |- 登录注册
  |- 游戏大厅
  |- 百家乐页面
  |- 骰子页面
  |- 钱包与记录页面

Backend API
  |- Auth 模块
  |- User 模块
  |- Wallet 模块
  |- Room 模块
  |- Bet 模块
  |- Game Engine 模块
  |- Settlement 模块
  |- History 模块
  |- Admin 模块

Infrastructure
  |- PostgreSQL
  |- Redis
  |- WebSocket Gateway
  |- Docker / Nginx
```

## 6. 前端设计

### 6.1 页面结构

前端第一阶段建议包含以下页面：

1. 登录页
2. 注册页
3. 游戏大厅页
4. 百家乐房间页
5. 骰子房间页
6. 钱包页
7. 下注记录页
8. 个人中心页

### 6.2 前端模块划分

#### Auth 模块

- 登录
- 注册
- Token 持久化
- 登录态校验

#### Lobby 模块

- 游戏列表展示
- 房间入口展示
- 当前游戏状态展示

#### Wallet 模块

- 余额显示
- 账变流水展示

#### Baccarat 模块

- 下注区域
- 发牌结果展示
- 倒计时展示
- 历史开奖记录

#### Dice 模块

- 玩法区域
- 开奖结果展示
- 倒计时展示
- 历史开奖记录

#### History 模块

- 用户下注历史
- 用户结算历史

- 截止下注后应锁定操作按钮
- 开奖或发牌后应即时更新结果和余额

## 9. 游戏设计

### 9.1 百家乐设计

#### 支持玩法

第一阶段建议只做以下基本玩法：

- 庄
- 闲
- 和
- 庄对
- 闲对

#### 单局流程

1. 创建新局
2. 进入下注倒计时
3. 玩家下注
4. 截止下注
5. 系统发牌
6. 根据规则补牌
7. 生成本局结果
8. 结算所有下注
9. 广播结果并开始下一局

#### 需要注意的实现点

- 发牌逻辑应由服务端控制
- 牌局结果与下注结算必须绑定同一个 `round_id`
- 结算时必须保证幂等，防止重复派奖
- 所有牌局结果都要可追溯

### 9.2 骰子游戏设计

#### 支持玩法

第一阶段建议只做以下玩法：

- 大
- 小
- 单
- 双
- 指定点数

#### 单局流程

1. 创建新局
2. 开启下注倒计时
3. 玩家下注
4. 截止下注
5. 服务端生成开奖结果
6. 按下注类型结算
7. 广播结果并开始下一局

#### 需要注意的实现点

- 每局开奖时间固定，适合做快节奏循环桌
- 骰子开奖结果应与本局数据绑定存档
- 所有结算逻辑必须在服务端完成

## 10. 游戏状态机设计

建议两种游戏都采用统一状态机，便于后端维护：

- `waiting`
- `betting`
- `closed`
- `drawing`
- `settling`
- `finished`

说明：

- `waiting`：等待新局开始
- `betting`：下注阶段
- `closed`：已停止下注
- `drawing`：开奖或发牌阶段
- `settling`：结算阶段
- `finished`：本局结束

## 11. 钱包与结算设计

### 11.1 钱包原则

- 钱包余额由服务端维护
- 前端只展示，不可信任前端余额
- 每次下注先校验余额
- 每次结算后必须记录账变流水

### 11.2 下注流程

1. 用户提交下注请求
2. 服务端校验登录态
3. 服务端校验当前局是否允许下注
4. 服务端校验余额是否足够
5. 扣减可用余额
6. 写入下注记录
7. 写入账变流水
8. 推送下注成功事件

### 11.3 结算流程

1. 服务端读取本局结果
2. 加载本局所有下注
3. 逐笔计算输赢
4. 更新下注状态
5. 增加中奖用户余额
6. 写入派奖流水
7. 标记本局已结算

### 11.4 幂等要求

必须防止以下问题：

- 同一局重复结算
- 同一笔下注重复派奖
- 网络重试导致重复扣款

建议措施：

- 对结算流程设置唯一事务锁
- 对账变流水的业务引用建立唯一约束
- 对下注请求使用请求幂等键

## 12. 实时通信设计

游戏页面建议采用 WebSocket 而不是纯轮询。

### 12.1 服务端推送内容

- 当前房间状态
- 当前局号
- 当前倒计时
- 本局已截止下注通知
- 本局开奖结果
- 当前用户余额变化

### 12.2 客户端处理要点

- 首次进入页面时先拉取一次初始状态
- 建立 WebSocket 后订阅房间频道
- 收到结果事件后更新开奖记录
- 断线重连后重新同步当前局状态

## 13. 安全设计

### 13.1 账号安全

- 密码必须使用 `bcrypt` 加密存储
- 登录接口应限制请求频率
- Token 需要设置有效期
- 重要操作需要服务端鉴权

### 13.2 业务安全

- 所有下注金额必须服务端校验
- 所有赔率必须服务端计算
- 所有开奖结果必须服务端生成
- 禁止通过前端参数直接影响结算

### 13.3 数据安全

- 钱包变动必须保留完整流水
- 记录关键操作日志
- 管理员操作应可审计

## 14. 管理后台设计

## 18. 风险与注意事项

### 18.1 合规风险

若平台涉及真实货币、充值、提现、代理分销或跨地区运营，将面临显著法律与合规风险。若未来要做真实资金业务，必须在产品设计前先完成法务与牌照评估。

### 18.2 技术风险

- 钱包与结算流程容易出现并发问题
- 实时通信断线重连会影响游戏体验
- 游戏状态同步不一致会导致前后端显示异常

### 18.3 产品风险

- 首版玩法过多会拖慢开发
- 后台能力不足时问题排查困难
- 若不先做好账变流水，后期很难补数据可信度

## 19. 当前建议

第一阶段的最佳策略是：

1. 先做虚拟筹码娱乐版
2. 先打通“登录 -> 大厅 -> 骰子 -> 结算 -> 历史”完整链路
3. 再接入百家乐
4. 最后再补后台和监控

这样可以更快验证整体架构是否稳定，也能降低早期开发复杂度。
